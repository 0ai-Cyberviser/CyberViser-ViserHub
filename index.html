Defensive honeypot telemetry

Collect real-world attack patterns without exposing production systems.

CyberViser honeypots should run on isolated disposable infrastructure, outside Google Workspace and outside production repos. The public website documents the pipeline; the sensors collect sanitized adversary behavior for detection engineering, fuzz corpus mutation, and Hancock training datasets.

A

Isolated decoy surface

Deploy low-interaction HTTP, SSH, API, GraphQL, and object-storage decoys on separate cloud hosts with no private credentials and no route to production.

B

Safe telemetry capture

Capture requests, headers, payload shapes, timing, user agents, paths, auth attempts, and exploit fingerprints. Redact secrets, truncate payloads, hash IPs when publishing.

C

PeachTree normalization

Normalize events into NDJSON/JSONL records with schema, source sensor, ATT&CK mapping, payload class, parser target, and dataset safety label.

D

PeachFuzz mutation

Convert observed payload structures into fuzz seeds for HTTP, JSON, GraphQL, YAML, archive, and log parsers, then generate minimized reproducers.

Guardrails

Do not run honeypots on the same host as GitHub Pages, Google Workspace, admin consoles, email, Drive, or production services. Do not publish raw IPs, credentials, malware binaries, or private victim data. Publish sanitized aggregates, schemas, and reproducible safe samples only.

Privacy boundary

Public site, private operations.

This domain is designed as a public landing hub. Administrative systems, Google Drive, Google Admin, private repositories, internal notes, raw honeypot logs, and sensitive infrastructure data stay off the public website.

  1. DNS verification: TXT challenge records are public DNS records by design.
